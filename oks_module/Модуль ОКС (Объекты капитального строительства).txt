Модуль ОКС (Объекты капитального строительства)
Документ подготовлен для уточнения логики реализации модуля ОКС строго в рамках утверждённого ТЗ «ОКС (Объекты капитального строительства)», без расширения функционала и договорных обязательств.

________________


1. Назначение модуля
Модуль ОКС предназначен для:
* учёта и сопровождения объектов капитального строительства;
* контроля этапов, сроков и статусов выполнения;
* фиксации проблем, комментариев и контрольных точек;
* хранения и систематизации документации по каждому объекту.
Модуль обеспечивает прозрачность контроля ОКС для руководства, бухгалтерии и контролирующих подразделений.
________________


2. Основные сущности системы
В модуле реализуются следующие сущности:
* Объект капитального строительства (ОКС)
* Этап объекта
* Документ (акты, чертежи, ТЗ, служебные записки и др.)
* Комментарий / проблема / контрольная точка
* Ответственный
* Инициатор строительства
* История изменений
________________


3. Инициатор строительства (обязательное уточнение)
Для каждого объекта капитального строительства обязательно указывается инициатор строительства.
Инициатор строительства:
* фиксируется при создании объекта;
* выбирается вручную (из списка пользователей / подразделений);
* отображается в карточке объекта и во всех сводках;
* используется:
   * для фильтрации объектов;
   * для аналитики и отчётности;
   * для прозрачности ответственности.
❗ Инициатор строительства:
* не равен ответственному за этап;
* не изменяется автоматически;
* может быть изменён вручную при наличии прав.
________________


4. Источники данных
Данные в модуль ОКС поступают из следующих источников:
* ручной ввод сотрудниками ОКС;
* документы и файлы, прикрепляемые вручную;
* при наличии интеграции — данные из 1С (факт выполнения, оплаты, статусы);
* внешние отчёты (при необходимости, вручную).
________________


5. Данные, вводимые вручную
Вручную вносятся:
* наименование объекта;
* инициатор строительства;
* этапы объекта;
* плановые сроки и бюджеты;
* ответственные лица;
* статусы этапов;
* комментарии и примечания;
* сопроводительные документы и файлы.
________________


6. Статусы объектов и этапов
Используются следующие статусы:
Для объектов:
* Планируется
* В работе
* Приостановлен
* Завершён
Для этапов:
* Планируется
* В работе
* Просрочен
* Завершён
Статусы обновляются:
* вручную ответственными;
* автоматически при наступлении контрольных условий (сроки).
________________


7. Автоматические проверки и подсветки
Система автоматически:
* подсвечивает просроченные этапы;
* предупреждает о приближении срока (−10 и −5 дней);
* фиксирует отсутствие обновлений по объекту > X дней;
* сохраняет историю изменений;
* визуально отображает план / факт (при наличии факта).
❗ Автоматические проверки не изменяют данные, а только информируют.
________________


8. Аналитика и агрегированные данные
В рамках ТЗ реализуется агрегированная аналитика:
* количество объектов в работе;
* количество просроченных этапов;
* отклонения по срокам;
* активность обновлений по объектам;
* распределение объектов по ответственным и инициаторам;
* сводка по статусам.
Без прогнозов и без ИИ-аналитики.
________________


9. Документооборот по объектам
* Документы загружаются и хранятся в системе;
* Каждый документ привязывается:
   * к объекту;
   * к этапу (при необходимости);
* Поддерживаются версии документов;
* Документы имеют статусы:
черновик / согласование / утверждён;
* Поиск и фильтрация по:
   * объекту;
   * типу документа;
   * дате;
   * автору;
   * статусу.
________________


10. Уведомления
Реализуются уведомления по событиям:
   * наступил срок этапа;
   * приближение просрочки (−10 / −5 дней);
   * этап просрочен;
   * объект не обновлялся > X дней;
   * объект или этап завершён.
Каналы: интерфейс (push), email — согласно ТЗ.
________________


11. Права доступа
Минимальная ролевая модель:
   * ОКС-admin — полный доступ;
   * Ответственный — редактирование своих объектов и этапов;
   * Инициатор — просмотр и комментарии по инициированным объектам;
   * Viewer — просмотр без редактирования.
Права на документы и объекты разграничиваются по ролям.
________________


12. Вау-эффекты в рамках ТЗ (делать смело)
⚡ Карточка объекта как единая панель: этапы, документы, статусы, комментарии
⚡ Цветовая индикация статусов и отклонений
⚡ Таймлайн изменений по объекту
⚡ Быстрые фильтры по инициатору, статусу, ответственному
⚡ Пустые состояния (“нет этапов”, “нет документов”, “нет обновлений”)
Все эффекты — визуальные и UX, без расширения логики.
________________


Иерархия и вложенность объектов ОКС
1. Общий принцип вложенности
В модуле ОКС поддерживается иерархическая структура объектов.
Объект капитального строительства может быть:
   * верхнего уровня (например, здание, корпус);
   * вложенным объектом (помещение, часть помещения, зона).
Поддерживается структура вида:
   * участок
   * Здание
├─ Помещение
│ ├─ Часть помещения
├─ Помещение
   * Отдельный объект без вложенности
Глубина вложенности:
      * минимум 2 уровня;
      * без жёсткого ограничения, но в рамках разумной иерархии (дерево).
________________


2. Связь «родитель — дочерний объект»
Для каждого объекта ОКС:
      * может быть указан родительский объект;
      * дочерний объект наследует контекст родителя (название, принадлежность, инициатора);
      * дочерний объект:
      * имеет собственные этапы;
      * собственные документы;
      * собственных ответственных;
      * собственные статусы.
❗ Родительский и дочерний объекты не сливаются:
      * статусы считаются отдельно;
      * документы хранятся отдельно;
      * сроки контролируются отдельно.
________________


3. Инициатор строительства при вложенности
      * Инициатор строительства задаётся на верхнем уровне объекта.
      * Для вложенных объектов:
      * по умолчанию наследуется от родителя;
      * может быть изменён вручную при наличии прав.
Инициатор отображается:
      * в карточке каждого объекта;
      * в агрегированных отчётах;
      * в фильтрах.
________________


4. UI-представление вложенности (обязательно)
В интерфейсе должна быть реализована наглядная иерархия объектов.
UI-требования:
      1. Древовидный список объектов
      * раскрытие / сворачивание уровней;
      * визуальный отступ для вложенных объектов;
      * индикация статуса на каждом уровне.
      2. Карточка объекта
      * отображается:
      * родительский объект (если есть);
      * список дочерних объектов;
      * быстрый переход между уровнями.
      3. Фильтрация и поиск
      * поиск по всем уровням;
      * фильтрация по инициатору / ответственному учитывает вложенные объекты.
      4. Сводки
      * в сводках по верхнему объекту отображается агрегированная информация по всем вложенным объектам:
      * количество этапов;
      * количество просрочек;
      * общее состояние (без автоматического изменения статусов).
________________


5. Правила редактирования и удаления
      * Нельзя удалить объект верхнего уровня, если у него есть дочерние объекты.
      * Перенос дочернего объекта к другому родителю возможен только вручную и при наличии прав.
      * Все операции с иерархией логируются (кто, когда, что переместил).
________________


6. Вау-эффекты в рамках ТЗ (без расширения логики)
⚡ Дерево объектов с цветовой индикацией статусов
⚡ Сворачивание/разворачивание зданий → помещений
⚡ Быстрый переход «вверх / вниз по иерархии»
⚡ Агрегированная подсветка проблем на уровне здания
⚡ Пустые состояния (“у объекта нет вложенных элементов”)
